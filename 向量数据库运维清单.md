## 1. Milvus（Docker Compose）

> Milvus 由 3 个容器组成，必须一起正常，向量检索才可用。

### 1.1 部署位置
- 目录: `~/milvus-standalone`
- 编排文件: `~/milvus-standalone/docker-compose.yml`
- 数据目录: `~/milvus-standalone/volumes/`
- 说明:
  - `docker-compose.yml` 是启动清单，定义了 Milvus/etcd/minio 三个服务。
  - `volumes/` 是数据持久化目录，重启容器后数据仍在。

### 1.2 服务组成
- `milvus-standalone`: Milvus 主服务（对外提供 `19530`）
- `milvus-etcd`: 元数据协调服务（Milvus 依赖）
- `milvus-minio`: 对象存储服务（Milvus 索引和数据依赖）
- 说明:
  - 只要其中任意一个异常，Milvus 可能连接失败或查询异常。

### 1.3 端口
- `19530`: Milvus gRPC（你的 Python 客户端主要连接这个）
- `9000/9001`: MinIO（Milvus 内部依赖，通常无需对公网暴露）
- `2379`: etcd（Milvus 内部依赖，通常无需对公网暴露）
- `9091`: Milvus 管理端口（内部监控/管理用途）
- 说明:
  - 对业务代码来说，通常只需要关注 `19530` 是否可用。

### 1.4 常用命令
```bash
# 进入 Milvus 部署目录（必须在这里执行 compose）
cd ~/milvus-standalone

# 后台启动所有服务（-d = detached，不占当前终端）
docker compose up -d

# 查看当前状态（重点看 STATUS 是否都是 Up）
docker compose ps

# 查看 Milvus 主容器实时日志（用于排错，Ctrl+C 退出）
docker logs -f milvus-standalone

# 重启整个 Milvus 服务组（配置变更后常用）
docker compose restart

# 停止服务（容器停止，但数据仍在 volumes 里）
docker compose stop
```

### 1.5 当前状态（手动填写）
- `docker compose ps` 结果: `<Up/Down>`
- `19530` 监听: `<是/否>`
- 判断标准:
  - `Up`: 说明容器进程存活
  - `19530` 监听为“是”: 说明客户端可尝试连接 Milvus

## 2. Chroma（systemd）

> Chroma 当前不是 Docker 运行，而是由 Linux 系统服务 `systemd` 托管。

### 2.1 部署位置
- 服务文件: `/etc/systemd/system/chroma.service`
- 数据目录: `/opt/chroma/data`
- 启动命令:
  - `chroma run --host 0.0.0.0 --port 8000 --path /opt/chroma/data`
- 说明:
  - `chroma.service` 决定了 Chroma 如何开机自启、崩溃重启。
  - `--path /opt/chroma/data` 表示数据落盘目录，不会因会话断开而丢失。

### 2.2 端口
- `8000`（HTTP API）
- 说明:
  - 你的 Chroma 客户端连接地址通常是 `http://服务器IP:8000`。

### 2.3 常用命令
```bash
# 查看服务状态（是否运行、最近日志摘要）
systemctl status chroma --no-pager

# 启动服务
systemctl start chroma

# 停止服务
systemctl stop chroma

# 重启服务（改配置后常用）
systemctl restart chroma

# 实时查看服务日志（排错最有用）
journalctl -u chroma -f
```

### 2.4 当前状态（手动填写）
- `systemctl status chroma`: `<active/inactive>`
- `8000` 监听: `<是/否>`
- 判断标准:
  - `active (running)`: 服务正在运行
  - `8000` 监听为“是”: Chroma API 对外可访问

## 3. 一次性盘点命令（服务器执行）

```bash
# ===== 1) 基础信息 =====
# 查看主机名（确认当前在哪台机器）
hostname

# 查看系统内核与版本（排查环境差异）
uname -a

# 记录当前时间（用于审计“何时盘点”）
date

# ===== 2) Milvus 状态 =====
# 进入 Milvus 部署目录
cd ~/milvus-standalone

# 输出当前目录，防止误在别处执行命令
pwd

# 列出目录内容，确认 compose 文件和 volumes 是否存在
ls -lah

# 查看 3 个服务是否都为 Up
docker compose ps

# 读取 Milvus 最近 50 行日志，快速看是否有报错
docker logs --tail 50 milvus-standalone

# ===== 3) Chroma 状态 =====
# 看 Chroma 服务状态（active/inactive/failed）
systemctl status chroma --no-pager

# 看是否开机自启（enabled/disabled）
systemctl is-enabled chroma

# 读取 Chroma 最近 50 行日志
journalctl -u chroma -n 50 --no-pager

# ===== 4) 端口总览 =====
# 查看关键端口是否在监听
# 8000 Chroma, 19530 Milvus, 9000/9001 MinIO, 9091 Milvus管理端口, 2379 etcd
ss -lntp | egrep '8000|19530|9000|9001|9091|2379'

# ===== 5) 进程/容器总览 =====
# 用表格方式查看所有容器（名字、镜像、状态、端口）
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}\t{{.Ports}}"

# 查 Chroma 进程（确认是否有重复实例）
ps -ef | grep -i "chroma run" | grep -v grep
```

盘点结果怎么判定:
- Milvus 正常: `docker compose ps` 全部 `Up` 且 `19530` 正在监听。
- Chroma 正常: `systemctl status chroma` 为 `active (running)` 且 `8000` 正在监听。
- 若两者都正常，你的双向量库即可认为“在线可用”。

## 4. 统一巡检（每次重启后执行）

```bash
# 检查 Milvus 服务组状态
cd ~/milvus-standalone && docker compose ps

# 检查 Chroma 服务状态
systemctl status chroma --no-pager

# 同时确认两个核心端口都在监听
ss -lntp | egrep '8000|19530'
```

说明:
- 这 3 条命令是你的“最小健康检查”。
- 每次服务器重启后，先跑这 3 条再跑业务代码。

## 5. 常见故障与处理

### 5.1 Chroma 报端口占用
- 现象: `Address 0.0.0.0:8000 is not available`
- 处理:
```bash
# 杀掉遗留手工进程（常见于之前前台启动过 chroma）
pkill -f "chroma run"

# 交给 systemd 正常重启
systemctl restart chroma
```
- 原因解释:
  - 同一个端口不能被两个进程同时监听。
  - 常见场景是“手工启动 + systemd 启动”冲突。

### 5.2 Milvus 不可用
- 处理:
```bash
# 进入 Milvus 目录
cd ~/milvus-standalone

# 后台拉起所有服务
docker compose up -d

# 检查是否全 Up
docker compose ps

# 查最近 100 行日志定位错误
docker logs --tail 100 milvus-standalone
```
- 原因解释:
  - Milvus 依赖 etcd/minio，任一组件异常都可能导致连接失败。
  - 所以先 `up -d`，再 `ps` 和 `logs` 是标准排查路径。

## 6. 代码连接规范
- Milvus: `MILVUS_HOST` + `MILVUS_PORT=19530`
- Chroma: `CHROMA_HOST` + `CHROMA_PORT=8000`
- 禁止在代码中硬编码公网 IP

说明:
- 推荐把变量写入环境变量或 `.env` 文件，代码里只用 `os.getenv()` 读取。
- 这样换服务器 IP 时只改配置，不改代码。

## 7. 定期清理清单（建议每周一次）

> 目标: 在不影响 Milvus/Chroma 正常运行的前提下，释放磁盘空间。  
> 原则: 先看占用，再清理；先清缓存日志，再碰业务目录。

### 7.1 你看到的 `overlay 21%` 是什么
- `df -h` 里的 `overlay` 不是单独分区，而是 Docker 容器的联合文件系统视图。
- 你看到多个 `overlay 21%`，本质都映射到底层同一块磁盘（如 `/dev/vda3`），不是重复占了多份 21%。
- 真正决定磁盘占用的是:
  - `/var/lib/docker`（镜像、容器层、缓存）
  - 业务数据目录（如 Milvus/Chroma 数据目录）

### 7.2 清理前检查（先执行）
```bash
# 看整体磁盘使用率
df -h

# 看 Docker 各项占用
docker system df

# 看大目录（根目录层面）
sudo du -h --max-depth=1 / 2>/dev/null | sort -hr | head -n 20
```

### 7.3 每周安全清理（推荐）
```bash
# 1) 清理 apt 缓存和无用依赖
sudo apt clean
sudo apt autoclean
sudo apt autoremove -y

# 2) 清理 systemd 日志，仅保留最近 7 天
sudo journalctl --vacuum-time=7d

# 3) 清理临时目录
sudo rm -rf /tmp/*
sudo rm -rf /var/tmp/*

# 4) 清理 Docker 未使用对象（不影响正在运行容器）
docker system prune -f

# 5) 如需进一步释放镜像空间（谨慎）
# 会删除未被任何容器使用的镜像
docker image prune -a -f

# 6) 清理未使用卷（谨慎）
docker volume prune -f
```

### 7.4 严禁误删目录（你的核心数据）
- `~/milvus-standalone/volumes/`（Milvus 数据）
- `/opt/chroma/data`（Chroma 数据）
- `~/milvus-standalone/docker-compose.yml`（Milvus 启动配置）
- `/etc/systemd/system/chroma.service`（Chroma 服务配置）

### 7.5 清理后复检（必须执行）
```bash
# 磁盘是否下降
df -h

# Docker 占用是否下降
docker system df

# 服务是否正常
systemctl status chroma --no-pager
cd ~/milvus-standalone && docker compose ps
ss -lntp | egrep '8000|19530'
```

### 7.6 建议的清理频率
- 每周: 执行 `7.3` 全流程（测试环境）
- 每月: 增加一次大目录排查（`du -h --max-depth=1 /`）
- 重大更新前: 先做快照/备份，再清理
